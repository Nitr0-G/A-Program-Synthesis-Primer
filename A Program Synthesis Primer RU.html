Что такое program synthesis?

В области языков программирования и верификации традиционная задача program synthesis включает в себя построение программы на основе высокоуровневой математической спецификации. Предположим, вы хотите написать программу, которая вычисляет факториал некоторого числа n>1. Вы начинаете с математического определения факториала, n!, и постепенно расширяете его — переписываете, используя свои знания о свойствах факториала и математики, — пока не дойдете до исполняемой программы.

Несмотря на высокие цели и богатые традиции, этот обобщающий подход, называемый deductive synthesis— имел ограниченный успех. Проще говоря, математические спецификации трудно написать, а преобразование спецификации в программу очень трудно автоматизировать.

Combinatorial synthesis

В последнее время исследователи изучают формы синтеза, которые проще автоматизировать. Стиль синтеза, на котором я сосредоточусь здесь, был впервые применен в проекте Sketch. Идея заключается в том, что мы начинаем с эталонной реализации p_naive. Думайте об этом как о наивном решении проблемы, которое, как вы уверены, работает правильно, но которое вы не будете внедрять в производство, возможно, потому, что оно довольно неэффективно. Ваша цель - найти программу p_smart эквивалентную p_naive, но более эффективную и более сложную для правильного написания кода.1

Для иллюстрации давайте рассмотрим чрезвычайно простой пример. p_naive приведена ниже, где для простоты мы предполагаем, что все целые числа являются битовыми векторами с 8-битной шириной.

int p_naive(int x):
    y = x * 2
    return y

Вы, как умный (но не очень сообразительный) программист, думаете, что вы могли бы добиться большего успеха, написав программу следующего вида, но вы не уверены, что поместить в??

int p_smart(int x):
    y = x << ?? // the operator << is shift left
    return y

В некотором смысле, p_smart представляет собой семейство программ — все возможные значения ??. Конечно, правильным завершением здесь является замена ?? на 1.

Automating synthesis

Итак, как мы автоматически находим значение ?? чтобы сделать p_smart эквивалентным по результатам p_naive. Легко! Существуют 28 разных экземпляров ??, потом мы будем подбирать разные значения до тех пор пока не сможем найти нужный. В общем, это приводит к комбинаторному "взрыву"(к очень большому кол-ву значений для перебора). Мы не можем избежать комбинаторной проблемы, но оказывается, что мы можем точно охарактеризовать пространство поиска наших экземпляров как формулу, которую мы можем дать эффективному готовому SMT solver'у - как мы увидим немного позже.

Во-первых, давайте попробуем постепенно формализовать проблему. Мы будем использовать переменную h для обозначения нашей неизвестности ??.

ОДЗ 1

Найдем значение* для h такое, чтобы чтобы для всех x, p_naive(x) == p_smart(x). Другими словами: мы найдем для h все значения, с которыми результаты двух p_ будут одинаковы

ОДЗ 2

Мы можем думать о h как о входных данных для p_smart. Итак, наша цель на самом деле состоит в том, чтобы найти значение для h такое, чтобы для всех значений x, p_naive(x) == p_smart(x,h).

ОД3 3

Теперь мы можем рассматривать программу, скажем, p_naive, как логическую связьφn(x,y) между её input и output переменными, x и y. Идея заключается в том, что отношение истинно тогда и только тогда, когда p_naive(x) == y.

Итак, наша цель действительно состоит в том, чтобы найти значение для h чтобы для всех значений x и y, у нас было значение φn(x,y)⟺φs(x,h,y) true.

Последнее ОДЗ

Наконец, мы готовы логически сформулировать проблему. Найдите удовлетворяющие значения для следующей формулы ∀x,y.φn(x,y)⟺φs(x,h,y)Обратите внимание, что переменная h свободна. Следовательно, любое удовлетворяющее значение этой формулы только содержится в h.

На данный момент мы полностью свели задачу синтеза к решению логической формулы. Мы можем сделать это с помощью SMT-решателя, такого как Z3 или bitwuzla(https://bitwuzla.github.io/).

Детальное рассмотрение примера выше

Теперь мы можем применить описанный выше процесс для кодирования p_smart и p_naive в виде формул. Мы закодируем их в теории битовых векторов first-order(первого порядка), которая содержит все стандартные операции, которые нас интересуют..

φn(x,y)≡y=x∗2 φs(x,y,h)≡y=x<<h

Теперь мы напишем их используя Z3 вместе с его Python API следующим образом:

Сначала мы определяем все переменные как 8-битные векторы.

Весь пример доступен тут: GitHub

x = BitVec('x',8)
y = BitVec('y',8)
h = BitVec('h',8)

Затем мы обозначем φn и φs:

phi_n = y == x * 2
phi_s = y == x << h

Теперь мы готовы вызвать главную часть Z3: 

# first, encode the universally quantified formula
# (the == symbol is if and only if)
encoding = ForAll([x,y], phi_n == phi_s)
# second, call Z3 and check if there is a model
s = Solver()
s.add(encoding)
s.check()
# print the model (i.e., the value of h)
print s.model()

Если ты "запустишь" этот код, то на выходе увидишь [h = 1] - как и ожидалось!

Синтез, основанный на тестировании(Test-driven synthesis)

Подход, представленный выше, работает над двумя программами: он пытается найти программу, эквивалентную некоторой эталонной реализации. Вместо того чтобы писать полноценную эталонную реализацию, мы сейчас просто напишем несколько тестовых примеров и найдем программу, которая пройдет все из них. Вы могли бы думать об этом как о тестовой разработке на стероидах!

Допустим, мы хотим написать функцию, которая, учитывая битовый вектор x, возвращает битовый вектор y, равный 1 в позиции первого 0 справа, x, и 0 где-либо ещё.2

Для примера, 

# test 1
x = 00000000
y = 00000001
# test 2
x = 00000011
y = 00000100
# test 3
x = 00000010
y = 00000001

Предположим, у вас есть следующая догадка о том, как эффективно написать такую программу: 

y = (~(x + ??)) & (x + ??)

Здесь ~ является битовым not и & является битовым and. Наша цель - найти решения для двух неизвестных , которые приведут к желаемой программе.

Как и прежде, мы пишем программу как логическое отношение:

φs≡y=(∼(x+h1)) & (x+h2)

Обратите внимание как мы написали две неизвестных - мы их написали как две переменные

Аналогично, мы можем написать тестовые примеры(test cases) как отношение:

φt≡t1∨t2∨t3

где

t1≡x=0∧y=1t2≡x=3∧y=4t3≡x=2∧y=1

Наконец, мы решаем следующую формулу: ∀x,y.φt⇒φsОбратите внимание, что у нас есть импликация(https://ru.wikipedia.org/wiki/Импликация), а не эквивалентность между этими двумя отношениями. Интуитивно понятно, что отношение, определенное тестовыми примерами, невелико и не определяет поведение, отличное от 3-х предоставленных нами тестовых примеров. Итак, то, что мы ищем, - это p_smart, в которой все тестовые примеры отображаются в ее отношении (или являются подмножеством).

Давайте напишем это используя Z3! Процесс будет аналогичен процессу выше

Код для этого примера также доступен: GitHub

phi_s = y == (~(x + h1)) & (x + h2)
t1 = And(x == 0, y == 1)
t2 = And(x == 1, y == 2)
t3 = And(x == 2, y == 1)
phi_t = Or(t1,t2,t3)

encoding = ForAll([x,y], Implies(phi_t, phi_s))

Решая формулу мы получаем следующий результат: [h2 = 1, h1 = 0], что как раз в рузльтате использования значений выше позволяет получить корректный результат нашей p_smart

Поскольку мы имеем дело с тестовыми примерами, предоставляемых нами тестовых примеров может оказаться недостаточно, чтобы заставить SMT-решатель найти нужную программу; в таком случае вы можете предоставить больше тестов. Например, если я просто дам Z3 3 тестовых примера(test cases), то я получу[h2 = 15, h1 = 14], что является неправильным результатом (проверьте это).

Подытожим

Мы рассмотрели комбинаторную форму синтеза программы, где мы определяем пространство поиска как программу с неизвестными. Вы можете задаться вопросом, как релизовать ещё и control-flow нашей p_smart. В этом документе это отлично описывается: Synthesis of Loop-free Programs.

Идея поиска в пространстве программ недавно была исследована в более широком спектре. Оказывается, передача задачи SMT-решателю - не всегда правильный путь; иногда пользовательский алгоритм перечисления превосходит SMT-решатели. Однако SMT-решатели очень хороши в поиске magic constants, до которых простое перечисление, возможно, никогда не дойдет. Итак, в задачах с перебором битов лучше всего использовать symbolic execution.
